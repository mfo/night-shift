## Ce qu'on a appris

### 1. `config.to_prepare` ne suffit pas pour invalider le cache ViewComponent
- L'initializer `view_component_dev_reload.rb` utilise `config.to_prepare` mais ce hook ne se déclenche que quand Rails détecte un changement sur un fichier `.rb` surveillé
- Supprimer/ajouter des `.haml`/`.erb` ne déclenche PAS le reload
- **Fix : `touch app/components/dsfr/input_status_message_component.rb`** — toucher le `.rb` du composant force Rails à recharger et `config.to_prepare` s'exécute
- Alternative envisagée : déplacer l'invalidation dans un `before_action` (garanti à chaque requête)

### 2. MCP Playwright : le `.mcp.json` ne suffit pas
- Le fichier `.mcp.json` à la racine du projet n'est PAS détecté automatiquement par Claude Code
- Il faut enregistrer le serveur MCP via la CLI : `claude mcp add playwright -- npx -y @playwright/mcp@latest`
- Le `--` est obligatoire pour séparer les args du serveur MCP des args de `claude mcp add`
- Après ajout, il faut relancer Claude Code (`/exit` puis `claude`)

### 3. Auth dev : utiliser letter_opener pour le lien sécurisé
- L'auto-login dev crée un user mais le `redirect_if_untrusted` redirige vers `lien-envoye` (trusted device check)
- Solution : naviguer sur `localhost:3000/letter_opener`, cliquer sur le lien de connexion sécurisé dans le 1er email
- Ceci valide le trusted device et redirige vers `/patron`

### 4. Screenshots : filtrer les éléments visibles
- `page.$$('.fr-alert')` peut matcher des éléments hidden → timeout au screenshot
- Toujours vérifier `await element.isVisible()` avant de capturer
