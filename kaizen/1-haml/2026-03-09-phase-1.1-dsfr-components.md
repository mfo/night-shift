## ⚠️ Ce Qui a Coincé

### Blocages Rencontrés

1. **Permission refusée pour suppression de fichiers HAML**
   - **Problème :** Impossible d'exécuter `rm` sur les fichiers .haml
   - **Cause :** Restrictions de permissions dans la configuration Claude Code
   - **Solution appliquée :** Création d'un script shell `delete-haml-phase-1.1.sh` que l'utilisateur a exécuté manuellement
   - **Temps perdu :** ~2 min

2. **Signature GPG requise pour commit**
   - **Problème :** Le commit initial a été bloqué car GPG signature attendue
   - **Cause :** Configuration git du projet requiert signature GPG par défaut
   - **Solution appliquée :** Utilisation de `git commit --no-gpg-sign` après accord utilisateur
   - **Temps perdu :** ~1 min

3. **Conversion HAML → ERB pour attributs Ruby dynamiques**
   - **Problème :** Syntaxe `%div{ **options }` en HAML nécessite `tag.attributes(**options)` en ERB
   - **Cause :** Différence de syntaxe entre HAML (hash attributes) et ERB
   - **Solution appliquée :** Utilisé `<div <%= tag.attributes(**options) %>>` dans notice_component
   - **Temps perdu :** ~2 min (réflexion sur la bonne syntaxe)

4. **❌ ERREUR CI : Arrays de classes non convertis correctement (callout_component)**
   - **Problème :** J'ai écrit `class="<%= callout_class %>"` mais `callout_class` retourne `["fr-callout", "fr-background-alt--blue-france"]` (array)
   - **Cause racine :** Je n'ai PAS lu le fichier Ruby du composant avant la conversion. Je me suis concentré uniquement sur le template HAML
   - **Impact :** 2 tests system specs échoués en CI (`.fr-callout` non trouvé car rendu `["fr-callout", ...]` au lieu de `"fr-callout ..."`)
   - **Solution :** `class="<%= callout_class.join(' ') %>"`
   - **Temps perdu :** ~5 min (détection via CI + correction + amend)
   - **Gravité :** 🔴 HAUTE - Casse des tests, erreur silencieuse en dev (pas de crash, juste HTML invalide)

5. **❌ ERREUR LINTER : Balise auto-fermante HTML5 (input_component)**
   - **Problème :** J'ai écrit `<input id="..." type="checkbox" />` avec le slash de fermeture
   - **Cause racine :** J'ai copié la syntaxe HAML `%input{ ... }/` qui génère du XHTML/XML. Le linter herb impose HTML5 strict
   - **Impact :** Échec linter herb en CI : `html-no-self-closing (1 offense)`
   - **Solution :** `<input id="..." type="checkbox">` sans le `/>`
   - **Temps perdu :** ~3 min (détection via CI + correction + amend)
   - **Gravité :** 🟡 MOYENNE - Bloque la CI mais facile à fixer

6. **❌ ABSENCE DE VALIDATION LOCALE AVANT COMMIT**
   - **Problème :** J'ai committé sans lancer `bundle exec rspec` ni `bun lint:herb` localement
   - **Cause racine :** J'ai suivi le prompt qui disait "commit + rapport" sans checkpoint de validation intermédiaire
   - **Impact :** Commit initial cassé, nécessite amend (git history moins propre)
   - **Temps perdu :** ~5 min (cycle CI complet avant détection)
   - **Gravité :** 🔴 HAUTE - Mauvaise pratique, coûteux en temps CI

7. **❌ ERREUR CI 2ÈME ITÉRATION : Espacement et guillemets HTML (alert_component)**
   - **Problème :** Après avoir fixé les 2 premières erreurs, 2 nouveaux tests ont échoué (`spec/lib/redcarpet/trusted_renderer_spec.rb`)
   - **Cause racine 1 - Espacement :** ERB préserve l'indentation du fichier source dans le HTML généré
     ```erb
     <div>
       <% if condition %>     ← Ces espaces sont rendus dans le HTML
         <h2>Title</h2>       ← Ces espaces aussi
       <% end %>
     </div>
     ```
     HAML compacte automatiquement, ERB non.
   - **Cause racine 2 - Guillemets :** HAML génère `class='...'` (simples) pour les attributs hash, ERB `class="..."` (doubles)
     ```haml
     %div{ class: "my-class" }  → <div class='my-class'>  (simples!)
     ```
     ```erb
     <div class="my-class">     → <div class="my-class">  (doubles)
     ```
   - **Impact :** 2 tests redcarpet échoués en CI (HTML généré différent du HTML attendu)
   - **Solution :**
     - Espacement : Utiliser `<%-` et `-%>` pour supprimer les espaces blancs
     - Guillemets : Utiliser `class='...'` au lieu de `class="..."`
   - **Temps perdu :** ~8 min (cycle CI + investigation + correction + amend)
   - **Gravité :** 🔴 HAUTE - Différence subtile, difficile à détecter sans lancer les tests

8. **❌ ERREUR CI 3ÈME ITÉRATION : Espacement dans download_component**
   - **Problème :** Après avoir fixé alert_component, 1 nouveau test a échoué (`spec/controllers/instructeurs/dossiers_controller_spec.rb:2062`)
   - **Cause racine :** Même problème d'espacement qu'alert_component, mais je n'avais appliqué la correction QUE sur alert_component, pas sur les autres fichiers DSFR
   - **Erreur détectée :** Test attendait `"Télécharger le fichier \ntoto.txt"` mais obtenait `"Télécharger le fichier \n    toto.txt"` (4 espaces en trop)
   - **Origine :** Les retours à ligne entre `<%= name %>` et `<span class="fr-download__detail">` étaient préservés par ERB
   - **Impact :** 1 spec instructeur/dossiers échoué en CI
   - **Solution :** Supprimer les retours à ligne avant `<% end -%>` et coller les balises:
     ```erb
     <%= truncate ? name.truncate(20) : name %><span class="fr-download__detail">
     <%= helpers.download_details(attachment) %>
     </span><% end -%>
     ```
   - **Note importante :** Ce n'était PAS un test flaky. Le test timeout en local (DB query cancellation) mais l'erreur CI était claire et reproductible
   - **Temps perdu :** ~5 min (investigation + correction + amend)
   - **Gravité :** 🟡 MOYENNE - Même type d'erreur qu'avant, aurait dû être anticipée et appliquée à tous les fichiers DSFR

## 💡 Learnings Clés

### Ce que j'ai appris sur CE projet

1. **Les composants DSFR n'ont pas de tests unitaires** - Ils sont testés via les system specs qui utilisent les vues
2. **Le projet utilise ViewComponent** (pas React Components) - Important de ne pas confondre
3. **Structure des composants :** `app/components/nom/nom.html.erb` + `nom.rb` + `nom.yml` (i18n)
4. **DSFR = Design System FR** - Framework CSS du gouvernement français, classes très spécifiques à préserver
5. **❌ CRITIQUE - Méthodes de classes retournent des arrays :** Plusieurs composants utilisent des méthodes qui retournent `["class-1", "class-2"].compact` au lieu de strings. HAML gère ça automatiquement, pas ERB.
6. **❌ Le linter herb est strict sur HTML5 :** Pas de balises auto-fermantes comme `<input />`, doit être `<input>`
7. **❌ CRITIQUE - ERB préserve l'indentation, HAML compacte :** ERB génère le HTML avec les espaces/newlines du fichier source. HAML compacte automatiquement. Il faut utiliser `<%-` et `-%>` pour supprimer les espaces.
8. **❌ CRITIQUE - ERB utilise doubles quotes, HAML simples :** HAML génère `class='...'` pour les attributs hash, ERB génère `class="..."`. Peut casser des tests qui comparent le HTML exact.

### Ce que j'ai appris sur l'IA & ce type de tâche

1. **❌ La conversion HAML → ERB N'EST PAS si simple** - Il y a des pièges subtils :
   - HAML gère automatiquement les arrays, ERB non
   - HAML génère du XHTML (balises auto-fermantes), ERB doit être HTML5 pur
   - **IL FAUT LIRE LE CODE RUBY, pas juste le template**
2. **❌ L'IA ne peut PAS être "fire-and-forget" sans validation locale** - Les erreurs silencieuses (arrays non joints) ne se voient qu'en prod/tests
3. **✅ TodoWrite essentiel pour les tâches par batch** - Permet de ne rien oublier et de montrer la progression
4. **❌ Batch de 12 fichiers TROP GRAND sans validation intermédiaire** - Aurait dû valider après 3-4 fichiers
5. **❌ Le prompt initial était INCOMPLET** - Manquait :
   - "Lire le fichier Ruby"
   - "Lancer le linter avant commit"
   - "Vérifier les arrays de classes"
   - "Pas de balises auto-fermantes"
6. **❌ Appliquer les corrections de façon SYSTÉMATIQUE** - Après avoir découvert le problème d'espacement sur alert_component, j'aurais dû vérifier et corriger TOUS les autres fichiers DSFR, pas juste celui qui a échoué
7. **❌ Ne pas confondre "test flaky" et "erreur réelle"** - Le test download_component qui timeout en local n'était PAS flaky : le message d'erreur CI était clair et reproductible (espacement), le timeout local était juste un problème de DB

### Hypothèses Validées / Invalidées

- ❌ **La conversion HAML → ERB peut être faite automatiquement** - NON, trop de subtilités (arrays, espacement, guillemets)
- ❌ **Le markup HTML reste identique** - NON, ERB génère un HTML différent (espacement, guillemets) même si visuellement identique
- ✅ **Les composants simples (< 50 lignes) se migrent en ~1-2 min chacun** - OUI pour la conversion brute, mais +23 min de validation/corrections pour 4 erreurs
- ❌ **Les tests passent après migration sans vérification** - NON, 5 specs échouées en 3 itérations (arrays + espacement × 2 + guillemets)
- ❌ **L'IA peut migrer sans supervision** - NON, nécessite validation locale systématique ET compréhension des subtilités HAML vs ERB
- ⚠️ **Le linter peut détecter certaines erreurs** - OUI (balises auto-fermantes) mais PAS les arrays ni l'espacement/guillemets
